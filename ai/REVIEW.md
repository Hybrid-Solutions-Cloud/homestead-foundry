## Should-fix

1. **Catalog-size discrepancy: SPIKE-01 says 680 calls, SPIKE-05/ADR-0006/the cost design use 340. Confirmed real, and the cap does not comfortably hold at the larger figure.**

   The source plan (`plans/source/mai-image-2-5-art-match.md`, sections 7 to 8) is unambiguous: "about 170 scenes x 2 candidates" is 340 total images, and "about 3 hours for 340 images" at 2 RPM. SPIKE-01 Q5 mis-cites this as "roughly 340 scenes at 2 candidates each (about 680 calls)," doubling the source figure by apparently reading "340 images" as "340 scenes" and then multiplying by 2 again. SPIKE-05 and ADR-0006 correctly use the source plan's 340-image basis (about 17 to 68 USD on MAI-Image-2.5). This is not a new finding: `design/performance-efficiency.md` section 5.1 and section 8 item 3 already caught it, state both numbers, name SPIKE-01 as the outlier, and say "reconcile the scene count against the prompt library before authorizing the backfill." That existing flag is correct and should stand.

   What is new here: I ran the cap check the design doc stops short of. At 680 calls, `performance-efficiency.md`'s own arithmetic gives 34 to 136 USD on MAI-Image-2.5 (its section 5.1 already shows this), or roughly 20 to 95 USD on Flash. Add the locked three-voice backfill (about 32 USD, independent of this question) and a full one-shot push lands at roughly 66 to 168 USD on 2.5, or 52 to 127 USD on Flash. The top of both ranges exceeds the 100 USD cap; ADR-0006's own table shows the 340-image basis landing "right at 100 USD" as the ceiling case, so doubling the image count plausibly breaks the ceiling rather than just approaching it. **The 100 USD cap does not clearly hold if the catalog turns out to be closer to 340 scenes than 170.**

   Recommend: before authorizing the image backfill (a separate, later, owner-gated step per MASTER-PLAN), count the actual scene entries in the brand-specific prompt library (`resources/Illustration_Prompts_All_Stories.md`). If the real count is near 170, no change is needed. If it is near 340, either raise the cap for that one month, split the backfill across two months, or default to Flash for the bulk pass (which keeps the worst case under 100 USD combined with the voice backfill). This does not affect Phase 8 infrastructure deployment, which only spends on two bounded smoke calls.

2. **`en-AU-<confirm-at-spike>:MAI-Voice-2` placeholder in `pipeline-integration-design.md` section 2.5, when the value is already known and cited.**

   SPIKE-02 Q2 already found and cited the exact voice: `en-AU-Lisa:MAI-Voice-2`, female, with the full style set including `excited`, sourced from the same Microsoft Learn page (`mai-voices`) the rest of the doc set cites. I independently re-fetched that page today and it is still the only en-AU MAI-Voice-2 voice listed, unchanged. Despite this, ADR-0003 decision 5, the design docs, and the `brand.json` code sample in `pipeline-integration-design.md` all continue to carry it as an open unknown to "confirm at spike time" or "read off the Foundry playground," and the code sample literally uses a placeholder token instead of the known value.

   This caution is not unreasonable on a preview surface (voice names can be renamed before GA, which the docs correctly worry about elsewhere), so I have not edited the placeholder myself; that is a substantive content call, not a typo. Recommend the docs cite `en-AU-Lisa:MAI-Voice-2` as the current, Learn-confirmed default and keep the live playground check as a drift guard rather than presenting the identifier itself as unknown.

3. **Minor procedural risk: the Foundry project may be more than optional at deploy time.**

   Microsoft Learn's MAI image how-to (`use-foundry-models-mai`) lists "A Microsoft Foundry project" as a deployment prerequisite, worded more strongly than this doc set's treatment of `proj-studioai-media-01` as fully optional and deferrable past the model-deployment step (`implementation-guide.md` step 4, C2: "may be deferred to the audition session without blocking the hold gate"). The same Learn page's own CLI deployment example, which `implementation-guide.md` W3 mirrors exactly, targets the account directly and never references a project, so the practical risk looks low. Recommend the deploy operator watch Gate 4 closely: if `az cognitiveservices account deployment create` or the Foundry portal path errors without a project present, create `proj-studioai-media-01` first (moving C2 ahead of W3) and note the reordering in `as-built.md`. Not a blocker; the runbook is already gated and reversible at this step.

4. **r2-upload.mjs reconciliation scope: correctly captured in the design doc, not yet reflected back into ADR-0008.**

   Confirmed directly against the live repos, not just against the docs: I read `tools/r2-upload.mjs` in both brand-specific consumer reader-app repos in full today: one has a 4-attempt backoff loop in `putObject`; the other has a single attempt, no retry, no loop. The drift is real and exactly as `pipeline-integration-design.md` section 0 and its "ADR gaps found" item 1 describe it: SPIKE-06 recorded the two files as identical (an error in that spike), the design-phase code re-audit caught the actual drift, and section 1 folds the fix into the same reconciliation commit as a one-way port (the retry loop ported from the repo that has it into the repo that lacks it). ADR-0008 itself only names the `publish.mjs` drift, not this one.

   As reviewer, I confirm the design's widening of the reconciliation scope is correct and low-risk: porting a retry loop onto a Windows-`wrangler`-abort workaround adds resilience and changes no behavior for successful uploads. Recommend a one-line addition to ADR-0008's Consequences or Follow-ups acknowledging the `r2-upload.mjs` retry-loop port, so the ADR and the design doc agree on what the reconciliation commit actually touches. This is documentation hygiene, not a functional gap, and does not block Azure infrastructure deployment.

Nothing found rises to a blocker for Phase 8 (the gated infrastructure runbook: resource group, account, deployment, roles, one vault secret, budget, and two bounded smoke calls). The four Should-fix items are real but sit on the far side of the MASTER-PLAN scope line: three concern the later, separately owner-gated bulk-generation work (catalog size versus the cap, the Lisa placeholder, the Foundry-project prerequisite watch-point at Gate 4), and one is a documentation-hygiene gap between ADR-0008 and the design doc in the consumer reader-app pipeline track, unrelated to the Azure create steps. The implementation guide's own confirmation-point and owner-input registers (P1 to P7, O1 to O7, W1 to W11) already correctly gate everything that needs a human decision before it happens; the Owner decisions list above maps the review agenda's asks onto that existing register rather than inventing new gates.
